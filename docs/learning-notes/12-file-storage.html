DAY 12 OF 18

Real File Storage

MinIO has been running in docker-compose.yml, completely unused, since Day 1. Today it finally does something - and along the way, five environment variables that were quietly being ignored since Day 1 start actually mattering.

The one-paragraph summary

Day 8 gave tasks "attachments" that were really just a name and a URL - a link to a file already sitting somewhere else (Drive, Dropbox, wherever). Today adds a SECOND way to attach something: a real uploaded file, streamed into the MinIO bucket that's been provisioned but untouched this whole time. Both kinds of attachment now live side by side on the exact same list, and look identical to the frontend - it never has to know or care which one it's looking at.

Same rule as always Every file involved has comments explaining what's new - especially lib/storage.ts (the one new file that talks to MinIO) and task.service.ts's toAttachmentSummary, worth reading closely.

Three new concepts, explained simply

1. A discovered gap: env vars that were silently ignored for 11 days

Digging in to build this, it turned out .env and .env.example already had five MinIO-related lines (MINIO_ENDPOINT, MINIO_PORT, MINIO_BUCKET, MINIO_ACCESS_KEY, MINIO_SECRET_KEY) sitting there since Day 1 - but config/env.ts's Zod schema never asked for them. A schema without .strict() just quietly ignores env vars it doesn't know about - so those five lines had been doing precisely nothing since the very first day of this project. Adding them to the schema is what finally makes them real.

2. Presigned URLs - a file our own server never touches on the way out

There's no route like GET /attachments/:id/raw streaming file bytes through our Express server. Instead, lib/storage.ts's getSignedDownloadUrl asks MinIO itself to generate a URL that's valid for exactly 15 minutes and points DIRECTLY at MinIO - the browser downloads straight from MinIO, our API never sees the bytes on the way out (it only saw them on the way IN, during upload). The expiry is baked into the signature itself, so a link that gets copy-pasted somewhere and forgotten just stops working on its own, with no code anywhere that has to actively revoke it.

3. "Exactly one of two fields" - a shape Mongoose doesn't enforce for you

TaskAttachment now has EITHER url (Day 8's link kind) OR storageKey/mimeType/sizeBytes (today's upload kind) - never both, never neither. Mongoose has no built-in "exactly one of these fields" validator, so this rule lives entirely in task.service.ts: createAttachment only ever sets url, createUploadedAttachment only ever sets the other three. A shared toAttachmentSummary helper checks which one is present and builds the right kind of url to send the frontend either way.

What actually happens, step by step

1
The server boots, connects to Mongo, then calls ensureBucketExists() - checks whether the "opssphere" bucket exists in MinIO yet, and creates it if not (a fresh MinIO container starts with ZERO buckets).
2
Someone opens a task and clicks the new "Upload a file" control - a plain, hidden <input type="file"> behind a styled label.
3
Picking a file immediately uploads it - a FormData multipart request, handled by multer on the way in, which hands the controller the raw bytes as a Buffer.
4
The API streams those bytes straight to MinIO under a namespaced key (orgs/<id>/projects/<id>/tasks/<id>/<uuid>-filename) and saves a TaskAttachment row pointing at that key.
5
The attachment shows up in the list right alongside any link-based ones, with its size shown ("142 KB") - same list, same styling, no visual distinction between the two kinds.
6
Clicking it downloads directly from MinIO via a signed URL good for 15 minutes; deleting it removes both the database row AND the object in MinIO.

A design detail worth understanding: why uploads don't crash the server on boot

ensureBucketExists() warns, it doesn't process.exit(1) config/env.ts crashes the ENTIRE server if ACCESS_TOKEN_SECRET is missing - correctly, since literally nothing works without auth. ensureBucketExists() deliberately does NOT follow that same pattern: if MinIO is unreachable, it logs a warning and lets the server keep starting. The reasoning: file uploads are ONE feature among many. A broken MinIO connection should mean "uploading a file fails with a clear error," not "tasks, tickets, the whole app is down." Matching failure severity to actual blast radius, not just copying the nearest existing pattern, is the real lesson here.

What was deliberately left simple for today

  • A 10 MB file-size limit, hardcoded in task.routes.ts - not user-configurable, a reasonable small default for a demo/teaching project.
  • No image thumbnails/previews - an uploaded image shows the same generic link + size as a PDF or a zip file. A reasonable "Day 17 polish" candidate, not today's scope.
  • No virus scanning / MIME-type allowlist - any file type multer accepts gets uploaded as-is. A real production deployment would want this; a teaching project demonstrating the CORE upload/ download loop doesn't need it to prove the concept.
  • Deleting an attachment's MinIO object is best-effort (wrapped in try/catch, logged, never re-thrown) - if MinIO happens to be down at that exact moment, the database row still gets removed rather than leaving the task stuck. A small orphaned object in MinIO is a much smaller problem than a task nobody can manage.

How to try it yourself

  1. Run pnpm install - THIS TIME there ARE new packages (@aws-sdk/client-s3, @aws-sdk/s3-request-presigner, multer).
  2. Make sure Docker's MinIO container is actually running (docker compose up -d minio if needed) - check http://localhost:9001 (the MinIO console, login opssphere / opssphere123) to confirm.
  3. Run pnpm dev, open a task, and use the new "Upload a file" control.
  4. Confirm the file appears in the MinIO console under the "opssphere" bucket, in a orgs/.../tasks/.../-shaped path.
  5. Click the attachment in the app to download it; wait 15+ minutes and try an OLD copied link - confirm it's expired.
  6. Delete the attachment from the app and confirm the object disappears from the MinIO console too.
