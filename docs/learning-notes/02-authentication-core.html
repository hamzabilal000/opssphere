Authentication Core

Register, verify your email, log in, log out. The first feature where the app actually does something you can see and use.

The one-paragraph summary

Today the app got its first real feature. You can create an account, get a verification email (delivered to Mailpit, our fake local inbox), click the link to verify it, log in, see a dashboard that proves you're actually logged in, and log out. Every one of these steps is real, working code — nothing is faked or mocked.

Same rule as Day 1 Every file involved has comments explaining what's new. This note is the map — the actual lesson is in the files themselves, especially auth.service.ts and auth.tokens.ts.

Three new concepts, explained simply

1. Password hashing — why we never store real passwords

We never save your actual password anywhere. Instead, lib/password.ts runs it through a one-way scrambler (bcrypt) and saves the scrambled result. "One-way" means you can turn a password into a scramble, but never turn the scramble back into the password. To check a login, we scramble the password you just typed and compare the two scrambles — we never "unscramble" anything.

2. JWTs and cookies — how the app remembers you're logged in

HTTP (the protocol behind every web request) has no memory by default — every request is treated like it's from a total stranger. To fix that, when you log in, the server hands your browser a small signed note (a "JWT") saying "this is user #123," and the browser automatically attaches that note to every future request via a cookie. The server checks the note's signature on each request (auth.middleware.ts) to confirm it's genuine and hasn't expired.

3. Email verification — proving you own that email address

When you register, we generate a long random code, email you a link containing it, and save only a scrambled version of that code in the database (same "one-way" idea as passwords). Clicking the link sends the code back to us; we scramble it the same way and check for a match. This proves you can actually receive mail at that address — not just that you typed something that looks like an email.

What actually happens, step by step

1
You fill out the register form. React sends your email + password to POST /api/v1/auth/register.
2
The backend checks the shape of your data. A shared Zod schema (used by BOTH frontend and backend) checks the email looks real and the password is at least 8 characters.
3
The backend hashes your password and saves a new User document in MongoDB, along with a hashed, expiring verification code.
4
An email gets "sent" — really it lands in Mailpit's inbox at localhost:8025, with a link back to your React app.
5
You click that link. It opens /verify-email?token=... in React, which immediately calls the backend to confirm the code and mark your account verified.
6
You log in. The backend checks your password's hash matches, confirms your email is verified, and sends back a signed JWT in a cookie.
7
You land on the dashboard. It calls GET /api/v1/auth/me — a "protected" route that only answers if your cookie checks out — and shows your account details back to you.
8
You log out. The backend clears the cookie. Trying to revisit the dashboard now bounces you back to login.

A security detail worth understanding: why login errors are vague

"Invalid email or password" — on purpose, both times Whether you typed an email that doesn't exist, or the right email with a wrong password, you get the exact same error message. If we said "no account with that email" separately from "wrong password," an attacker could use that difference to figure out which email addresses have accounts on the platform, one guess at a time — called account enumeration. One identical message for both cases closes that door.

How to try it yourself

  1. Make sure Docker is still running (pnpm docker:up if you restarted your computer).
  2. Run pnpm install again first — three new libraries were added (bcryptjs, jsonwebtoken, nodemailer).
  3. Run pnpm dev, open http://localhost:5173.
  4. Register with any email/password (8+ characters).
  5. Open http://localhost:8025 (Mailpit), open the email, click the verification link.
  6. Log in with the same email/password, and confirm you land on a dashboard showing your account info.

A real bug we hit (and what it taught us)

"Verification failed" even though it actually worked While testing this, clicking the email link showed "This verification link is invalid or has expired" — even though the account really was verified. The cause: React's development mode deliberately runs page-load effects twice on purpose, as a built-in check for exactly this kind of mistake. Our verification call isn't safe to run twice — the first call correctly "uses up" the token (like a one-time password reset link), so the second call fails. Because the failing second call finished last, its error message overwrote the real success message on screen.

The fix (already applied in VerifyEmailPage.tsx): a useRef flag that remembers "I already ran this" across both of React's intentional double-calls, so the API only actually gets hit once. This is a well-known React pattern, not a one-off hack — any page-load effect that calls an API which "consumes" something (a token, an invite, a one-time code) needs this same protection.
