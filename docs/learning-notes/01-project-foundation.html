TypeScript in 2 minutes (for someone who knows JavaScript)

Your other projects use plain JavaScript with require() and module.exports. This project uses TypeScript, which is just JavaScript with a few extra bits sprinkled on top. Here's everything new you'll actually see, in one place:

What you'll seeWhat it means, in plain English
import x from "y"Same job as const x = require("y") — just newer syntax.
export const x = ...Same job as module.exports = { x }.
: string, : number"This value must always be this type." Purely a hint for your editor — it's removed automatically before the code runs.
interface Thing { ... }A form template describing what fields an object must have. Doesn't create any real data — just a rule your editor checks for you.
field?: stringThe ? means "this field is optional, it's fine to leave it out."
SomeType<T>A "fill in the blank" type. E.g. ApiResponse<HealthStatus> means "a response, where the data is specifically a HealthStatus."
value as SomeType"Trust me, treat this as this type" — used when TypeScript can't automatically figure out what something is.
value?.field"Optional chaining" — try to read .field, but don't crash if value is missing.
You don't need to memorize this table Every time one of these shows up in the actual code, there's a comment right next to it explaining it again in context. Come back to this table only when you forget what a symbol means.

What actually happens when you run the app

Instead of a wall of separate explanations, here's the story in order — this is the exact path a request takes, start to finish:

1
You run pnpm dev. This starts two programs at once: the Express server (backend) and the Vite dev server (frontend).
2
The backend checks its settings first. apps/api/src/config/env.ts reads your .env file and double-checks every value is present and correctly formatted. If something's missing, it stops immediately with a clear error instead of starting broken.
3
The backend connects to MongoDB. apps/api/src/lib/db.ts opens the database connection — same job as the one-liner you already know, just in its own file.
4
The backend starts listening. apps/api/src/index.ts builds the Express app (from app.ts) and calls .listen(4000) — same as your usual app.listen(8080, ...).
5
You open the browser at localhost:5173. React loads, and App.tsx immediately calls the backend's health-check route to ask "are you alive?"
6
The backend replies. health.routes.ts sends back a small JSON object saying "yes, I'm running, and here's how long I've been up."
7
The screen updates. The card on screen flips from "Checking API connection…" to showing the real status — proving the whole chain (React → Express → MongoDB) is actually wired together correctly.

The folders, in one picture

OpsSphere/
├── apps/
│   ├── api/     → the Express backend (like your usual "server" folder)
│   └── web/     → the React frontend (like your usual "client" folder)
├── packages/    → small bits shared between api and web
└── docker-compose.yml → starts MongoDB + a few other local tools

The 4 folders inside packages/ are small and boring on purpose:

  • shared-types — just describes what API responses look like, so both sides agree.
  • validation — shared rules for "is this data valid?" (used by both sides eventually).
  • eslint-config and tsconfig — shared settings files, nothing to read closely yet.

What to actually do now

  1. Open the project in your code editor.
  2. Start with apps/api/src/index.ts — it's the shortest file and the comments walk through the startup order step by step.
  3. Then read apps/api/src/app.ts, then apps/web/src/App.tsx — in that order.
  4. Follow the run instructions below and confirm you see the green "status: ok" card in your browser.

How to run it

# 1. install dependencies for the whole project
pnpm install

# 2. start MongoDB, Redis, Mailpit, MinIO (Docker Desktop must be open)
pnpm docker:up

# 3. create your local settings file
copy .env.example apps\api\.env

# 4. run backend + frontend together
pnpm dev

Then open http://localhost:5173. A green "status: ok" card means everything is wired together correctly.
