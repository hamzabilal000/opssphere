True Drag-and-Drop Reordering

Since Day 8, dropping a card anywhere in a column always sent it to the very end of that column - dropping it back among cards that were already there just wasn't possible. Today makes the board do what it always looked like it could do: land a card in the EXACT spot you dropped it.

The one-paragraph summary

Every task's position field has tracked its spot in its status column since Day 8 - but the only thing that ever wrote to it was "count how many cards are already here, put the new one after all of them." Today's whole job is teaching moveTask a second trick: given an exact target slot, shift only the cards that actually need to move, and land the dragged card precisely there - whether that's a different spot in the SAME column, or an exact slot in a DIFFERENT one. No new model, no new permission, no new route - the same PATCH .../move endpoint from Day 8 just accepts one new optional field.

Same rule as always Every changed file has comments explaining what's new - especially task.service.ts's moveTask, which is where basically all of today's real thinking happened.

Three new concepts, explained simply

1. Moving one card is really "shift a range, not rewrite everything"

The naive way to reorder a column would be to fetch every task in it, rebuild the whole array in memory, and write every position back. Today does something narrower: since positions are always kept as a contiguous 0..N-1 sequence per column (an invariant every prior day already maintained without realizing it), moving a card from slot 2 to slot 4 only actually changes the cards that were SITTING between those two slots - everything else keeps its position untouched. That's two plain MongoDB updateMany calls with a $gt/$lte (or $gte/$lt) range and a $inc, not a loop over every task in the column.

2. Same column vs. different column are genuinely different problems

Reordering WITHIN one column is a classic "move an array element from index i to index j" operation - everything strictly between the old and new slot shifts by exactly one, in whichever direction closes the gap. Moving to a DIFFERENT column is two separate half-problems stacked back to back: close the gap left behind in the OLD column (everything after the old slot shifts up by one), then open a gap at the requested slot in the NEW column (everything from that slot onward shifts down by one). moveTask branches on oldStatus === newStatus right at the top specifically because these aren't the same math, even though they look like one feature from the board's point of view.

3. A card is now a valid drop target, not just the column background

Day 8's board only listened for a drop on the empty column <div> itself - fine when "drop anywhere in this column" always meant the same thing (the end). Today, each individual TaskCard button also has its own onDragOver/onDrop handlers: dropping directly ON a card means "land right before this one." The column's own drop handler still exists for the empty space below the last card (or an empty column entirely) and means "land at the end" - Day 8's only behavior, now just one of two ways to trigger a move instead of the only one.

What actually happens, step by step

1
Someone picks up a card in "In Progress" and drags it directly over a DIFFERENT card sitting two slots below it, in the same column.
2
Dropping there fires that target card's onDropBefore handler, which calls handleDrop("in_progress", targetCard's index, e) - the same column, a specific slot.
3
moveTask sees oldStatus === newStatus, computes insertAt (clamped to a valid slot), and shifts only the cards strictly between the old and new position by one each via a single updateMany.
4
The dragged task's own position is set to insertAt and saved - one document write for the moved card, one batched range-update for everyone it displaced.
5
Day 9's existing live-update wiring picks this up automatically - TASK_MOVED already invalidates the board's task list on every connected browser, no new socket code needed for today's feature at all.
6
Dragging a card to a DIFFERENT column and dropping it ON a specific card there works the same way, just taking the "moving to a different column" branch instead - close the old column's gap, open the new column's gap at that exact slot.

A design detail worth understanding: why two $inc calls instead of a loop

The contiguous-positions invariant is what makes this cheap This whole approach only works because every prior day (accidentally) kept column positions as a clean 0..N-1 sequence with no gaps - Day 8's original "append to the end" logic already guaranteed that. Because that invariant holds, moving one card only ever requires shifting the cards STRICTLY BETWEEN its old and new slot - never anything outside that range, and never anything in a column the card isn't touching. That's what turns "reorder a column" from an O(N) rewrite into two small, targeted range updates. If some future feature ever let positions drift out of that clean sequence (say, a bulk-import that assigns arbitrary numbers), this logic would need a "renumber the column first" step before it could trust the shortcut again.

What was deliberately left simple for today

  • No visual "drop indicator line" showing exactly where a card will land mid-drag - the cursor just shows the normal browser drag affordance. A polish-phase (Day 17) addition, not required to prove the underlying reorder logic works.
  • Dropping a card directly on itself is a harmless no-op API call (its computed target slot equals its own current slot) rather than being specially detected and skipped client-side - simpler code, no visible difference to the user.
  • No keyboard-accessible way to reorder cards (arrow keys, a context menu) - HTML5 drag-and-drop remains mouse-only, same limitation Day 8 already had. Day 16 "Hardening" is the natural home for an accessible alternative.

How to try it yourself

  1. Run pnpm install - no new packages today.
  2. Run pnpm dev, open a project's board with at least 3-4 tasks in one column.
  3. Drag a card from the bottom of a column and drop it directly ON a card near the top - confirm it lands exactly there, and everything in between shifts down by one slot, not just to the end.
  4. Drag a card into a DIFFERENT column and drop it on a specific card there (not the empty space below) - confirm it lands at that exact slot in the new column too.
  5. Open the same board in a second browser tab and confirm the reorder shows up there too, live, via Day 9's existing socket wiring - no new code path for that.
