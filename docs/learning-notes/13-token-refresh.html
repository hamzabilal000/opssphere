Automatic Access-Token Refresh

Since Day 3, a 15-minute access token has meant "logged in for 15 minutes" in practice - the app never actually used the 30-day refresh token sitting right next to it. Today closes that gap, entirely on the frontend, with zero backend changes.

The one-paragraph summary

Day 3 built a real access/refresh token pair: a short-lived 15-minute access token and a 30-day refresh token that rotates every time it's used. But nothing ever called POST /auth/refresh automatically - so in practice, anyone using the app got logged out every 15 minutes regardless of that 30-day refresh token quietly sitting unused in a cookie. Today teaches the frontend's single fetch wrapper to notice an expired-token 401, silently refresh, and retry the original request - so the 30-day session finally means something to the person using the app, not just to the database.

Same rule as always Every changed file has comments explaining what's new - especially lib/api.ts, where the entire feature lives. Read withAutoRefresh and refreshAccessTokenOnce closely; they're short, but every line is doing something deliberate.

Three new concepts, explained simply

1. A frontend-only day - no backend files changed

The backend route POST /api/v1/auth/refresh has worked correctly since Day 3: give it a valid refresh-token cookie, it rotates the refresh token and issues a new access token. The gap was never in the backend - it was that nothing on the frontend ever called it. That makes today unusual: every other day so far touched both apps/api and apps/web; today is apps/web only.

2. One shared wrapper, so every request gets the fix for free

Every API call in the app already funneled through one function - apiRequest (JSON calls) and apiUpload (file uploads, since Day 12). Both now build on a shared withAutoRefresh helper: try the request, and if it fails with the AUTHENTICATION_REQUIRED code specifically (not just any error), call /auth/refresh once and retry the original request exactly one time. Because every page and every hook already goes through these two functions, not one line of any actual page component needed to change to get this behavior.

3. The "thundering herd" problem - and the one-line fix

A real risk: if five different queries all fire at once when the access token has just expired (very normal - a dashboard loading orgs, projects, tasks, and tickets all at the same moment), each one would otherwise trigger its OWN call to /auth/refresh. Since refresh tokens ROTATE on every use (Day 3's design), the first refresh call would succeed and the second would then be using an already- rotated, now-stale refresh token - failing for no good reason. The fix is a single module-level variable, refreshInFlight: the first 401 starts the refresh and stores its promise; every other 401 that arrives while that promise is still pending just awaits the SAME promise instead of starting its own. One refresh call serves however many requests failed at once.

What actually happens, step by step

1
A user has been active for 20+ minutes. Their 15-minute access token has quietly expired, but their 30-day refresh token is still fully valid.
2
They click into a project. apiRequest fires the request; the backend correctly rejects it with 401 AUTHENTICATION_REQUIRED.
3
withAutoRefresh catches that specific error code, and calls refreshAccessTokenOnce() - which either starts a new POST /auth/refresh call, or (if one is already in flight from some other request) just waits on that same one.
4
The refresh succeeds; a new access-token cookie is set. withAutoRefresh retries the ORIGINAL request exactly once.
5
The retried request now succeeds - the user never sees an error, a redirect, or even a loading flicker beyond the request's normal latency.
6
If the refresh itself fails instead (the 30-day session is truly over - expired or revoked), the ORIGINAL 401 error is what gets thrown, not the refresh's error - and ProtectedRoute sends them to /login, same as before today.

A design detail worth understanding: why the error that surfaces matters

The ORIGINAL error wins, not the refresh's error When a refresh fails, withAutoRefresh deliberately re-throws the request's original 401 - not whatever error the failed refresh call produced. This matters because the refresh endpoint's own error (say, SESSION_REVOKED) isn't what any calling code was ever written to expect from, say, getOrganization(). Every existing catch site in the app already knows how to handle "this request failed" - throwing the original error means none of them need to change today, even though the reason it ultimately failed is new.

What was deliberately left simple for today

  • Only ONE retry attempt per request - if the retried request fails again for any reason, that second failure is what the caller sees. No retry loops, no exponential backoff.
  • No proactive/background refresh (e.g. a timer that refreshes a few minutes before the access token would expire) - today's approach is purely reactive: wait for a 401, then refresh. Simpler, and correct, at the cost of one extra round-trip the first time any token actually expires.
  • No user-visible "session expiring soon" warning - the whole point of this feature is that the refresh is invisible when it succeeds. A future day could add a warning specifically for the case where the REFRESH is about to fail (i.e., near the 30-day mark).

How to try it yourself

  1. Run pnpm install - no new packages today, this is just for consistency.
  2. Run pnpm dev, log in, and open your browser's DevTools → Application → Cookies.
  3. Manually delete just the access-token cookie (leave the refresh-token cookie alone) - this simulates the access token having expired naturally.
  4. Click around the app - notice nothing breaks or redirects to /login. Check the Network tab: you should see one request 401, then a POST /auth/refresh, then the original request repeated and succeeding.
  5. Now delete BOTH cookies (simulating the full 30-day session being over) and click around again - this time you should land on /login, same as before today.
