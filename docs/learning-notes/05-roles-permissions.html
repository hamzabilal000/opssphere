Roles & Permissions

Access is controlled by real, testable permission checks — not by hiding a button on the frontend and hoping nobody notices it's still reachable.

The one-paragraph summary

Day 4 gave every organization exactly two levels of access: "owner" or "member," hardcoded. Today replaces that with a real permission system - named permission strings (like role.manage), roles that are just a name plus a list of those strings, and a backend check on every sensitive action that actually enforces it. Along the way, organizations also gained Departments and Teams (simple groupings), and invitations now assign a real role instead of just creating a bare account.

Same rule as always Every file involved has comments explaining what's new - especially tenant.middleware.ts's new requirePermission function, which is the actual enforcement point for everything described below.

Three new concepts, explained simply

1. Permissions are just strings - roles are just lists of them

A permission is nothing more than a short label for ONE specific action, e.g. member.invite or role.manage. The full list lives in one place - PERMISSIONS in packages/shared-types - so backend, frontend, and validation all agree on the exact same strings. A role is just a name plus an array of these strings. Checking "can this person do X?" always comes down to one question: "is X in their role's permissions array?" There's no hierarchy, no inheritance, no "deny" rules layered on top - the SRS explicitly says to skip those for the MVP, and a flat allow-list is genuinely enough for what OpsSphere needs right now.

2. System roles vs. custom roles

Every new organization automatically gets two roles the moment it's created: Owner (every permission that exists) and Member (no permissions - a safe, do-nothing baseline). These are "system roles" - the code refuses to let anyone rename or delete them, so an org can never end up with zero usable roles. Anything beyond those two is a custom role - an admin picks a name and checks off whichever permissions they want, and the system treats it exactly the same way as a built-in one everywhere else (role assignment, permission checks - there's no special-casing).

3. Membership now points at a Role, not a fixed label

Day 4's Membership stored role: "owner" | "member" directly - a fixed, two-choice field. Today it stores roleId, pointing at a real Role document instead. This one change is what makes custom roles possible: a Membership doesn't need to know or care whether the role it points to is "Owner," "Member," or some custom role an admin invented five minutes ago - the lookup works identically either way.

What actually happens, step by step

1
You open an organization's details and see its roles - "Owner" and "Member" already exist automatically.
2
You create a custom role - say, "Support Agent" - checking two permissions: member.invite and team.manage.
3
You assign that role to a member using the role dropdown next to their name - this calls PATCH /organizations/:id/members/:membershipId.
4
That member tries an action their role DOES allow (inviting someone) - requirePermission("member.invite") checks their role's permissions array, finds a match, and lets the request through.
5
That member tries an action their role does NOT allow (deleting a role, which needs role.manage) - the exact same middleware checks the array, finds no match, and rejects with a clean 403. This is the Day 5 acceptance test, working end to end.
6
You invite someone new from inside the organization, picking a role for them up front - accepting that invitation now creates BOTH their account AND their Membership with that exact role, in one step.

A design detail worth understanding: two layers of protection, stacked

requireOrgMembership, then requirePermission - never just one Every write route in organization.routes.ts chains THREE middlewares: requireAuth (are you logged in), requireOrgMembership (are you a member of THIS org), THEN requirePermission(...) (does your specific role allow THIS action). Each one depends on data the previous one set on the request - requirePermission reads req.membershipPermissions, which only requireOrgMembership sets, which is why the order in each route definition matters and can never be shuffled.

What was deliberately left simple for today

  • No deny-rules or permission overrides - a flat allow-list is all the SRS asks for at this stage, and it's genuinely enough for what exists so far.
  • No role editing (rename or change permissions on an existing custom role) - only create and delete. Renaming can wait; deleting and recreating covers the same need for now.
  • Inviting an email that already has an OpsSphere account (into a SECOND organization) isn't supported yet - createOrgInvitation explicitly rejects this with a clear error rather than silently doing the wrong thing. A reasonable Day 6+ improvement.
  • The frontend hides admin-ish buttons based on a simple guess (your role isn't literally named "Member") - not a real permission check. That's fine, on purpose: see the SRS principle quoted in the code, "hiding a button on the frontend is a UX nicety, never real security." The backend is what actually enforces everything.

How to try it yourself

  1. Run pnpm install (packages/validation gained a new workspace dependency today - worth re-running even though no new npm packages were added).
  2. Run pnpm dev, log in, open an organization's details.
  3. Create a custom role with just member.invite checked, and assign it to a second test account (invite yourself with a "+alias" email, e.g. you+test@gmail.com, and accept it in an incognito window).
  4. Log in as that second account, open the same org, and try creating a role or department - confirm you get a clean "you don't have permission" error, since only member.invite was granted.
  5. Back in your original account, create a Department and a Team, and assign the team to that department.
