DAY 4 OF 18

Organizations & Multi-Tenancy

Every company gets its own isolated workspace — and today's whole job is proving that isolation is real, not just assumed.

The one-paragraph summary

Up to now, OpsSphere only knew about individual users. Today it learns about organizations — company workspaces — and the rule that everything from here forward has to follow: a user logged into one company's workspace can never see or touch another company's data, even if they try to. Everything else in the app (projects, tickets, teams — starting Day 5 onward) will belong to an organization and lean on the isolation rule built today.

Same rule as always Every file involved has comments explaining what's new — especially tenant.middleware.ts, which is the single most important file added today.

Three new concepts, explained simply

1. Multi-tenancy — many companies, one app, zero cross-over

"Tenant" is just a fancy word for "one company's workspace." Multi-tenancy means many tenants share the exact same running server and the exact same database — there's no separate copy of the app per customer. The upside is one codebase to maintain; the risk is that a bug could accidentally let one company see another's data. Today's whole job is closing that risk off at the ground floor, before any real business data (projects, tickets, etc.) exists to leak.

2. Organization vs. Membership — the workspace vs. your place in it

An Organization is the company itself (name, slug, time zone, business hours) — it doesn't know or care about any particular person. A Membership is the separate record that says "this specific user belongs to this specific organization, with this role." One person can have memberships in several organizations; one organization has many memberships. This is the exact same idea as a join table if you've used SQL before — just its own MongoDB collection here.

3. The golden rule — never trust an organization id from the client

Every organization-scoped URL looks like /api/v1/organizations/:organizationId/.... Here's the catch: that :organizationId is just text in a URL — anyone could edit it in their browser to a different company's real ID while still logged in as themselves. If the server just believed whatever ID showed up in the URL, that would be a serious data leak. Instead, EVERY request to one of these URLs is checked against a real Membership row in the database first — "does this exact user actually belong to this exact organization?" — before anything else in the request is allowed to run. See tenant.middleware.ts's requireOrgMembership function for exactly how.

What actually happens, step by step

1
You fill out "Create a new organization" on the dashboard — just a name and a slug (the short, URL-safe id, e.g. acme-inc).
2
The backend checks the slug is free. Slugs are unique across the whole app — MongoDB itself would also reject a duplicate as a backstop, even if two people tried to grab the same one at the exact same instant.
3
The Organization is created, and a Membership row is created right alongside it, making you its first owner.
4
The dashboard lists every org you belong to, with your role in each one shown next to it.
5
Clicking "View members" calls GET /organizations/:id/members — a request that goes THROUGH the tenant-context middleware, proving you're actually a member before any member list is returned.
6
The isolation test: if you were somehow logged in to Org A and tried editing the URL to Org B's real ID, the membership lookup for (you, Org B) simply finds nothing, and you're rejected with a 403 — before any Org B data is ever read.

A design detail worth understanding: two different route "shapes"

Not every organization route needs the same protection POST /organizations and GET /organizations (create one, list mine) only need "are you logged in at all" — there's no specific org to check membership against yet. GET /organizations/:organizationId and its /members route need a SECOND check: "are you logged in AND a member of THIS one." That's why organization.routes.ts chains two middlewares — requireAuth, then requireOrgMembership — only on the routes that have a specific organization in the URL.

What was deliberately left simple for today

  • Roles are just owner / member for now — the real permission system (named permission strings like project.create, bundled into custom roles) is Day 5.
  • No way yet to remove a member, promote/demote someone, or leave an organization — those need the role rules Day 5 introduces first (e.g. "the last owner can't demote themselves").
  • No "switch organization" concept — if you belong to several orgs, you see all of them on the dashboard at once; there's no single "active" organization to switch between yet.

How to try it yourself

  1. Run pnpm install (no new packages today, but harmless to re-run).
  2. Run pnpm dev, log in.
  3. On the dashboard, create an organization (pick any name/slug).
  4. Click "View members" — you should see just yourself, listed as owner.
  5. Create a second organization with a different slug — confirm both show up, each with you as owner, and that each one's member list only ever shows that one org's members.
