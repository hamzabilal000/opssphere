Frontend App Shell

Turning the backend into something that feels like a real product — a sidebar, an organization switcher, and no more testing everything through Postman.

The one-paragraph summary

Days 2-5 built real, working features, but every page was its own island - a centered white box on a plain background, with zero shared navigation between them. Today gives the app an actual shell: a sidebar with an organization switcher, a topbar, and pages that live INSIDE that layout instead of being full-screen forms. Two tools that were installed back on Day 1 but never used until now - TanStack Query and Zustand - finally do their real jobs.

Same rule as always Every file involved has comments explaining what's new - especially lib/queries.ts (TanStack Query) and store/activeOrgStore.ts (Zustand), since both are genuinely new CONCEPTS, not just new features.

Three new concepts, explained simply

1. TanStack Query - fetching data without writing useEffect every time

Every page since Day 2 wrote the same pattern by hand: useState for the data, useState for loading, useEffect to kick off the fetch, and a manual "refresh" function to call again after a create/delete. TanStack Query does all of that for you: useQuery({ queryKey, queryFn }) gives back { data, isLoading, isError } directly, and useMutation handles the create/update/delete side, including automatically re-fetching the right data afterward via invalidateQueries - see any function in lib/queries.ts for a real example.

2. Zustand - a tiny global store for state that ISN'T server data

"Which organization is the sidebar switcher currently showing" isn't something that comes from the API - it's just a fact about what this one browser tab is currently looking at. Zustand (store/activeOrgStore.ts) is a small, global piece of state ANY component can read or change directly, without passing it down through props layer by layer. It's deliberately a different tool from TanStack Query, because it's solving a different problem - "what is the UI currently doing," not "what does the server say."

3. Nested routes - two reusable layout layers instead of one page per URL

Look at App.tsx: <Route element={<ProtectedRoute />}> wraps a GROUP of routes, and everything inside only renders if you're logged in. Nested one level deeper, <Route path="/dashboard" element={<AppShell />}> wraps another group, and everything inside THAT renders inside the sidebar+topbar layout. Both layers use the exact same trick - a component that renders <Outlet /> wherever the actual nested page should appear. Any future page just gets added as one more line inside these wrappers, and gets both the login check AND the shell automatically.

What actually happens, step by step

1
You log in and land on /dashboard - the Overview page, now living inside the sidebar+topbar shell instead of alone on a blank background.
2
The sidebar's organization switcher loads your orgs via useOrganizationsQuery() and auto-selects the first one if nothing was picked before.
3
You click "Organization" in the sidebar - navigates to /dashboard/organizations/:id, the same roles/departments/teams/members management Day 5 built, now its own real page.
4
You create a role or invite someone - the mutation succeeds, a toast pops up in the corner confirming it, and the relevant list refreshes itself automatically (no manual refresh function needed - that's invalidateQueries at work).
5
You close the tab and come back later - the sidebar remembers which organization you had selected, because Zustand's persist middleware saved it to your browser's storage.
6
You log out from the topbar (works from ANY page now, not just one Dashboard) - TanStack Query's cache is cleared entirely, so no stale data from your account lingers around.

A design detail worth understanding: shadcn/ui, by hand

Why Button.tsx isn't an npm package The SRS names shadcn/ui as the component library. shadcn isn't installed like a normal package - its whole idea is that its CLI COPIES small component source files straight into your project, so you own and can edit the code directly instead of depending on a black box. That CLI needs an interactive setup step this environment can't run - so Button.tsx, Card.tsx, and Input.tsx do the same JOB by hand: small, editable, Tailwind-based components with the same "variant" idea shadcn's components use. Functionally equivalent; just written directly instead of scaffolded by a tool.

What was deliberately left simple for today

  • Command palette, keyboard shortcuts, and a notifications drawer are explicitly Day 17 polish, not today - see the SRS.
  • The org switcher is a plain <select>, not a fancy dropdown component - functional, not fancy, for now.
  • No dark mode, no responsive breakpoint tuning beyond what Tailwind gives for free - real accessibility/responsive polish is a Day 16 "Hardening" task per the roadmap.

How to try it yourself

  1. Run pnpm install - two new packages today (lucide-react for icons, zustand for the org-switcher state).
  2. Run pnpm dev, log in.
  3. Notice the sidebar, topbar, and organization switcher - try switching organizations if you have more than one.
  4. Create a role or invite someone from the Organization page and watch the toast notification appear in the bottom-right corner.
  5. Refresh the page (or close and reopen the tab) - the sidebar should remember which organization you had selected.
