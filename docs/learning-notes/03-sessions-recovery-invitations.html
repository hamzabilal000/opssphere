Sessions, Recovery & Invitations

Where you're logged in, what happens if a login token leaks, forgetting your password, and inviting someone new. Four features, all built on top of Day 2's login system.

The one-paragraph summary

Day 2 gave you a login that lasts until the cookie expires. Today makes that login system production-grade: you can see every device you're logged in on and kick one out, a stolen login token stops working automatically instead of staying valid for weeks, you can reset a forgotten password by email, and you can invite someone else to create an account without them registering themselves. Same rule as always — every file has comments, this note is the map.

Four new concepts, explained simply

1. Two tokens instead of one — access tokens vs. refresh tokens

Day 2 gave you a single login token (JWT) that lasted a while. The problem: the longer a token lives, the longer it stays dangerous if it's ever stolen. Day 3 splits this into two: a short-lived access token (15 minutes — this is what actually gets checked on every request) and a long-lived refresh token (30 days — its only job is "let me get a new access token without typing my password again"). When the access token expires, the browser quietly calls POST /auth/refresh using the refresh token to get a fresh one. You stay logged in, but the token that's actually being checked everywhere only has a 15-minute shelf life if it ever leaks.

2. Refresh rotation — why the old refresh token gets thrown away every time it's used

Every time /refresh is called, the server doesn't just check the refresh token — it immediately replaces it with a brand new one and makes the old one permanently invalid. This is called "rotation." The payoff: if someone ever steals a copy of your refresh token (say, from a compromised laptop) and your real browser keeps using the app normally, your browser will rotate the token away before the thief gets a chance to use their stolen copy. And if the thief tries their copy first — see the next concept.

3. Reuse detection — turning a stolen-token attempt into an alarm

Because every refresh token becomes invalid the moment it's rotated, there's exactly one way an "old, already-rotated-away" token could ever be presented again: someone has a copy that isn't the real, current one. So when that happens, we don't just reject the request — we revoke the entire session immediately, logging that device out completely. A legitimate user's browser would never naturally do this; only a stolen copy would. See refreshSession in auth.service.ts for exactly how this check works.

4. One-time tokens, reused three ways

Email verification (Day 2), password reset, and invitations all use the exact same underlying trick: generate a long random code, email the raw version, save only a scrambled (hashed) version in the database. Reusing one well-tested pattern for three different features, instead of inventing three slightly-different ones, is why auth.tokens.ts now calls this a generic "one-time token" instead of a "verification token."

What actually happens, step by step

Logging in and staying logged in

1
You log in. The backend creates a Session document (which device, which IP, when it expires) and sends back BOTH an access token cookie and a refresh token cookie.
2
The refresh cookie is scoped narrowly. It's only ever sent to /api/v1/auth/* routes (not every API call) — see the path option in auth.controller.ts.
3
15 minutes later, the access token expires. The next protected request would normally fail — this is where a future day wires up an automatic call to /auth/refresh.
4
/auth/refresh checks the refresh token against the Session's stored hash, rotates it, and returns a new pair of cookies.
5
On the dashboard, you see every active session — GET /auth/sessions — and can revoke any one of them, or log out everywhere else at once.

Forgetting your password

1
You request a reset from /forgot-password. The backend always replies with the same neutral message, whether or not that email actually has an account.
2
If it does exist, a one-time reset link is emailed (to Mailpit locally), expiring in 1 hour.
3
You click it, set a new password on /reset-password. The backend checks the code, updates your password hash, and — importantly — revokes every one of your active sessions, everywhere.

Inviting someone

1
A logged-in user invites an email from the dashboard. A 7-day invitation link is emailed to them.
2
They open the link at /accept-invitation?token=..., which shows their email and asks them to choose a password.
3
Submitting creates their account — pre-verified (accepting the link already proves they own that email) — and logs them straight in.

A security detail worth understanding: why logout revokes the session server-side

Clearing a cookie isn't the same as ending a session Day 2's logout only cleared the browser's cookie. That's enough for the browser you clicked "log out" on, but if a copy of the refresh token existed anywhere else (another tab's storage, a proxy log, a stolen backup), it would still work. Day 3's logout also marks the Session document itself as revoked in the database — so even a token that still physically exists somewhere stops being accepted the moment you log out.

How to try it yourself

  1. Run pnpm install (no new packages today, but harmless to re-run).
  2. Run pnpm dev, log in as usual.
  3. On the dashboard, look at "Where you're logged in" — you should see one row marked "This device."
  4. Log in again from a different browser (or incognito window) with the same account, then refresh the first dashboard — you'll see two sessions, and can revoke the second one from the first.
  5. Try "Forgot your password?" from the login page, and check localhost:8025 (Mailpit) for the reset email.
  6. Invite yourself with a second email address (or a "+alias", e.g. you+test@gmail.com) from the dashboard, then open the invite email and accept it.
