Polish

Day 6's learning note named this day explicitly: "command palette, keyboard shortcuts, and a notifications drawer are explicitly Day 17 polish, not today." Eleven days later, today is that day.

The one-paragraph summary

Three features, one theme: making the app feel faster to actually USE, not adding new business data. A real notifications system (backend model + two triggers - @mentions and task assignment - plus a bell icon with a live-updating unread badge) tells people when something needs their attention without them needing to go looking for it. A command palette (Cmd/Ctrl+K) lets anyone jump straight to an organization, a project's board, or a static page by typing a few letters, instead of clicking through the sidebar every time. Both ride on infrastructure earlier days already built - Day 9's Socket.IO connection and Day 6/15's existing organization/project queries - rather than needing anything fundamentally new.

Same rule as always Every changed file has comments explaining what's new - especially notification.service.ts, which documents a deliberate break from this project's usual "controller decides when to broadcast" pattern.

Three new concepts, explained simply

1. A model that belongs to a PERSON, not to one organization's data

Every model so far (Task, Ticket, Risk, ...) is scoped to ONE organization's URL - /organizations/:organizationId/.... A `Notification` breaks that pattern on purpose: the same logged-in person might have notifications from SEVERAL different organizations at once (a natural fit alongside Day 15's multi-org membership), so its list endpoint is mounted at its own top-level path, /api/v1/notifications, the same way Day 3's /auth/sessions is about the logged-in person rather than any one organization.

2. A second kind of Socket.IO "room" - one per USER instead of one per project

Every broadcast since Day 9 has gone to a PROJECT room, joined only when someone's actually looking at that board. A notification needs to reach someone no matter which page they're on - so lib/socket.ts gained a second room concept, userRoom(userId), that every socket joins AUTOMATICALLY the instant it connects (no explicit "join" event needed, unlike `join-project` - we already know the user's id from the same auth check every socket goes through). emitToUser(userId, ...) is the new counterpart to emitToProject(...).

3. Breaking the "controller decides when to broadcast" rule, on purpose

Every prior day's pattern: the SERVICE returns data, the CONTROLLER decides to call emitToProject(...) afterward. notification.service.ts's createNotification does its OWN emitting instead, right where the row gets created - a deliberate exception, not an oversight. Reasoning: a notification IS the broadcast; there's no separate "HTTP response" for it the way a task update has one. Threading "by the way, also notify these 3 people" back through task.controller.ts would be more plumbing for no real benefit over just handling "create it and tell them" as one atomic step in one place.

What actually happens, step by step

1
Someone comments "@bob@example.com can you check this?" on a task. task.service.ts's createComment detects the mention (Day 9's existing logic) and calls the new notifyMentions helper.
2
notifyMentions calls createNotification, which writes one Notification row AND immediately calls emitToUser(bobsUserId, "notification:created", ...) - both in the same step.
3
If Bob's browser is open (on ANY page, not just this task's board), his socket is sitting in his own user room already - the event arrives instantly, and Topbar.tsx's useNotificationSocket hook invalidates the notifications query.
4
The bell icon's unread badge updates without Bob refreshing anything. Clicking the bell opens a dropdown listing his notifications; clicking one marks it read and navigates him straight to that project's board.
5
Separately: anyone presses Cmd/Ctrl+K anywhere in the app. CommandPalette.tsx (mounted once in AppShell.tsx) opens, already showing every organization, every project in the currently-active organization, and a handful of static pages - typing filters the list, Enter jumps to the first match.

A design detail worth understanding: message text is frozen at creation time

A notification is a small, permanent record - not a live view Notification.message and linkPath are both computed ONCE, the moment the notification is created, and never recalculated. If the task that triggered a notification later gets renamed (or even deleted), the notification still reads exactly as it did the moment it fired - "You were assigned to task 'Fix login bug'" doesn't quietly become "Fix login bug (renamed)" or start pointing at nothing. This is the same instinct as Day 11's `riskScore` being computed fresh every time (never stored) taken in the OPPOSITE direction on purpose: a risk score should always reflect current reality, but a notification should always reflect what was true at the moment it happened - two different kinds of data, two deliberately different rules.

What was deliberately left simple for today

  • Only two notification TRIGGERS (mention, assignment) - ticket assignment/status changes, invitation acceptance, and risk ownership are all reasonable candidates for a THIRD+ trigger later, following the exact same createNotification(...) call shape.
  • The command palette only searches the ACTIVE organization's projects, not every project across every organization someone belongs to - switching organizations first (via the sidebar) is the reasonable trade-off against firing N extra network requests every time Cmd+K is pressed.
  • No arrow-key navigation within the command palette's results list - typing to filter, then clicking (or pressing Enter for the top match) covers the main use case without needing to track a selected-index state.
  • No dedicated keyboard-shortcuts help screen - the sidebar's small "⌘K to jump anywhere" hint is the one shortcut worth teaching; a fuller shortcuts reference is a reasonable future addition if more shortcuts get added.

How to try it yourself

  1. Run pnpm install - no new packages today.
  2. Open a task, comment @someone@example.com (a real project member's email) - log in as that person in a second browser and watch the bell badge update live, with no refresh.
  3. Assign someone else to a task - same live badge update for them.
  4. Click the bell, click a notification - confirm it marks read and navigates to that project's board.
  5. Press Cmd+K (or Ctrl+K) anywhere in the app - type part of a project or organization name, press Enter.
