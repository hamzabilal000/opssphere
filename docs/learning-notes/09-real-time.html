DAY 9 OF 18

Real-Time

Everything built so far asks the server a question and waits for one answer. Today the server learns to speak first.

The one-paragraph summary

Days 1-8 are entirely "request/response" - your browser asks, the server answers once, done. Today adds a second, genuinely different shape of communication: a connection that stays open, so the server can push a message to your browser WITHOUT you asking first. Two features ride on top of it: comments can finally be edited and deleted (with an "(edited)" label), and typing @someone@example.com in a comment tags them. But the headline feature is what happens when someone else, on another browser, moves a card on the SAME board you're looking at - you see it move too, instantly, with no refresh.

Same rule as always Every file involved has comments explaining what's new - especially apps/api/src/lib/socket.ts, since "a connection that stays open" is a genuinely new concept, not just a new feature built the same way as every day before it.

Three new concepts, explained simply

1. WebSockets vs. HTTP - a connection that stays open

Every request you've built until now opens a connection, sends one thing, gets one answer back, and closes - even "real-time-feeling" apps you've built before probably worked by re-asking the server every few seconds ("polling"). Socket.IO instead keeps ONE connection open between your browser and the server for as long as the page is open, over which either side can send a message to the other AT ANY TIME. The server "pushing" you a message it wasn't asked for is the entire new capability today unlocks.

2. Rooms - broadcasting to exactly the right audience

A connected socket isn't listening to anything by default - it has to "join a room" first, which is really just a string name (here, project:<id>). When a task moves, the server broadcasts to that ONE room, not to every connected browser everywhere - someone looking at a totally different project's board never hears about it. See projectRoom()/emitToProject() in lib/socket.ts.

3. The socket handshake gets the EXACT same auth check as every HTTP route

There's no separate "socket login." The very same access token cookie requireAuth checks on every HTTP request (see auth.middleware.ts) gets read and verified during the socket connection handshake too, in io.use(...) - a Socket.IO "middleware" that runs before a connection is even accepted, the same idea as Express middleware. And just like requireOrgMembership re-checks a real Membership row before trusting an organization id in a URL (Day 4's golden rule), the join-project handler re-checks a real Membership row before letting a socket into a project's room - a stolen or guessed project id can't be used to eavesdrop on a board you don't actually have access to.

What actually happens, step by step

1
You open a project's board. TaskBoardPage connects a socket and asks to join that project's room - a small "Live" badge in the header confirms it worked.
2
A teammate, on a different browser, drags a card on the SAME board.
3
Their move succeeds normally - same PATCH .../tasks/:id/move call as Day 8, same 409 rule if the task still has open subtasks.
4
The server broadcasts task:moved to the project's room, right after the database write succeeds.
5
Your browser receives it and quietly tells TanStack Query "that data's stale, go re-fetch it" - the exact same cache-invalidation mechanism Day 6 built for your OWN mutations, just triggered by someone ELSE's action this time. The card moves on your screen with no page refresh.
6
Meanwhile, in a comment, you type @teammate@example.com looks good - the mention gets detected, highlighted, and the comment can be edited or deleted later (by you, or anyone with task.manage).

A design detail worth understanding: why mentions are just "@" + an exact email

No autocomplete picker today - on purpose A real "@" mention experience needs a dropdown that filters project members as you type, arrow-key navigation, and inserting the right token into the textarea at the cursor position - genuine, fiddly frontend UI work that deserves its own focused effort and its own testing, not something to bolt onto an already-large Real-Time day. Instead, typing @ immediately followed by a project member's EXACT email (e.g. @hamza@example.com) is all detectMentions in task.service.ts looks for. It's less convenient, but completely unambiguous, fully server-verified against real project members, and the highlighting you see in the comment afterward proves it worked. A real autocomplete picker is a reasonable follow-up UI improvement on top of this same backend - the hard part (detecting and storing real mentions) is already done.

What was deliberately left simple for today

  • No notifications drawer, no "you were mentioned" badge anywhere in the UI - the SRS explicitly reserves a notifications drawer for Day 17 polish. Today's mentions are visible (highlighted in the comment itself) but don't push a separate alert anywhere yet.
  • No comment replies/threading - a nested comment tree is a bigger feature than fits alongside everything else today, and nothing so far has specifically promised it for this day.
  • Live updates only cover tasks and comments - attachments and time entries still just refresh the normal way (re-open the panel, or wait for your next own action to invalidate the cache). Extending the same room-broadcast pattern to those would be a small, mechanical follow-up.
  • The frontend reconnects a dropped socket automatically (Socket.IO's default behavior), but there's no "you're offline, reconnecting…" banner beyond the small Live/Offline badge - deeper connection-state UX is a reasonable future polish item, not built today.

How to try it yourself

  1. Run pnpm install - three new packages today (socket.io and cookie on the backend, socket.io-client on the frontend).
  2. Run pnpm dev, log in, open a project's board - you should see a teal "Live" badge next to the project name within a second or two.
  3. Open the SAME board in a second browser (or an incognito window, logged in as a different member of the same organization).
  4. Drag a card in one window - watch it move in the other, with no refresh.
  5. Open a task, post a comment mentioning a real project member's email (@their-email@example.com) - confirm it highlights, then edit and delete your own comment.
