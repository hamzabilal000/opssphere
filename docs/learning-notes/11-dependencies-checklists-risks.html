Dependencies, Checklists & Risk Register

Three features the Day 8 code comments explicitly deferred ("should have, not today's job") finally land - two as new fields ON the task you already know, one as a genuinely separate new module.

The one-paragraph summary

Task dependencies and checklists both live as new fields directly on the Task document - no new models, no new top-level collections, same "keep it embedded" instinct Day 8 used for position. A risk register, on the other hand, is a genuinely different kind of data (a project can have risks that never touch any one task), so it gets its own model and module - but PROJECT-level, not org-level like Day 10's Ticket, since a risk belongs to one project's plan, not the whole company. The one real new piece of logic today is a graph-walk: stopping someone from making task A depend on task B depend on task A.

Same rule as always Every file involved has comments explaining what's new - especially task.service.ts's assertNoDependencyCycle and risk.service.ts's LEVEL_SCORE lookup, both worth reading closely.

Three new concepts, explained simply

1. Cycle detection - the one genuinely new algorithm in the project so far

Every other "does this make sense" check in this codebase has been a simple lookup: does this id exist, does it belong to this project, is this person an active member. Dependencies need something sharper: if task A already depends on task B, task B can NEVER be allowed to depend on task A (even indirectly, through a longer chain like B → C → A). assertNoDependencyCycle walks FORWARD through the dependency graph starting from the candidate new dependencies - if that walk ever reaches the task you started from, it's a cycle, and the request gets a clean 400 instead of silently creating a chain of tasks that can never all be "done."

2. A checklist with genuinely NO ownership concept

Every other collaborative list in this app (comments, attachments, time entries) tracks WHO added each entry, and uses that for an "ownership-or-permission" edit/delete rule. Checklist items track none of that - any active project member can add, check off, rename, or remove ANY item, full stop. A checklist is meant to feel like a shared, disposable to-do list living on the task, not a set of records someone owns - a deliberate simplification, not an oversight.

3. A risk register that's PROJECT-level, unlike Day 10's org-level Ticket

Ticket (Day 10) has no projectId at all - a whole-company helpdesk concern. Risk has BOTH an organizationId AND a projectId - the same shape as Task/Sprint, not Ticket, because a risk ("the vendor might miss their delivery date") is naturally scoped to one project's plan. It's also governed differently: risk.manage is a FLAT permission with no ownership exception at all (unlike Ticket, where filing and editing your own ticket needs nothing special) - tracking a risk's severity and mitigation plan is treated as a coordinator-level activity, not something whoever happened to notice the risk should freely edit.

What actually happens, step by step

1
Someone opens a task on the board and scrolls to the new "Dependencies" section - a row of toggleable chips, one per other task in the project.
2
They mark it dependent on another task that isn't done yet - a "Blocked" badge appears immediately next to the section heading.
3
Dragging that card to "Done" is rejected with a clear 409 - the same shape as Day 8's "unfinished subtask" block, just one layer wider.
4
Anyone on the project adds a few checklist items underneath - "Write tests," "Update docs" - and checks them off as they go, no permission needed.
5
Meanwhile, on the project page, a manager opens the new "Risks" tab and logs "Vendor API has no SLA" with likelihood: high, impact: medium.
6
The register shows a computed score (6, out of 9) next to it, and lets them assign an owner and move its status from "identified" toward "mitigating."

A design detail worth understanding: reusing TASK_UPDATED instead of a new event

No CHECKLIST_CHANGED event was added to SOCKET_EVENTS Day 9 built a whole real-time layer with its own canonical event-name catalog. It would have been easy to add a fourth event just for checklist changes - instead, every checklist mutation emits the EXISTING TASK_UPDATED event with the whole current task. Since checklist items live embedded ON the task document, "a checklist item changed" and "the task changed" are the same fact from a listener's point of view - the board's existing live-update wiring picks it up automatically, with zero frontend socket code added today.

What was deliberately left simple for today

  • Checklist items can't be reordered by dragging - they render in the order they were created, same "append only" spirit as Day 8's task board columns (true drag-and-drop reordering is still an open, disclosed future upgrade for both).
  • Dependencies are only editable through the existing task update endpoint, not a dedicated one - dependsOnTaskIds is just one more optional field on updateTaskSchema, kept out of task CREATION on purpose (a brand-new task rarely has known dependencies yet).
  • The risk register has no comments, attachments, or Socket.IO live updates - reusing any of Day 8/9/10's richer machinery here would be a reasonable, mechanical follow-up, not something today's three-feature scope also needs to absorb.
  • riskScore uses a simple 1-3 per level, multiplied (max 9) - not a fancier weighted formula. Good enough to sort/prioritize by, and it's computed fresh every time, never stored, so it can never drift out of sync with the two fields it comes from.

How to try it yourself

  1. Run pnpm install (no new packages today - everything reuses what's already installed).
  2. Run pnpm dev, log in, open a project's board, open any task.
  3. Toggle another task on as a dependency, then try dragging this card to "Done" - confirm the 409.
  4. Mark the dependency itself "Done," then retry the drag - it should succeed now.
  5. Add a couple of checklist items, check one off, delete another - confirm the x/y counter updates.
  6. From the project page, click "Risks," add one, and confirm the score/status controls work as an Owner-role account (and that a plain Member can't see the create form or edit controls).
