DAY 10 OF 18

Support Tickets

A permission string that's been sitting unused since Day 5 finally gets a real module built around it - and regular "Member"-role users get something of their own to do.

The one-paragraph summary

Two loose threads get tied off today. First: ticket.assign has existed in the PERMISSIONS catalog since Day 5, always "reserved" for a support-tickets module that didn't exist yet - today it does. Second, and arguably more important: every module built so far (Projects, Tasks) is something a "Member"-role user can mostly only WATCH, not really own - creating one needs project.create or task.manage, permissions a plain Member doesn't have by default. Tickets flip that: ANY active member can file one. It's the first module in the app that gives a regular user a reason to open it themselves, not just a manager.

Same rule as always Every file involved has comments explaining what's new - especially ticket.service.ts, which has TWO different ownership-or-permission rules living side by side (one for editing, a different one for status changes) - worth reading closely.

Two new concepts, explained simply

1. Org-level, not project-level - a genuinely different shape than Task

Every model since Day 7 has hung off a Project (projectId on Task, Sprint, ...). A Ticket has NO projectId at all - just organizationId, one level up, the same depth as Organization itself. "The printer is broken" or "I can't log into the VPN" isn't naturally part of any one project - it's a whole-company helpdesk concern. ticket.routes.ts is mounted at the exact same base path as organization.routes.ts, not nested under projects the way task.routes.ts is.

2. Two DIFFERENT ownership rules on the same model

Every "ownership-or-permission" rule before today (Day 8's attachments/time-entries, Day 9's comments) used the same definition of ownership: whoever CREATED the thing. Tickets need two different answers depending on the action:

  • Editing details (title/description/priority): ownership means the ticket's creator - the person who reported the issue can still fix a typo in their own report.
  • Changing status (open → in progress → resolved → closed): ownership means the ticket's current assignee - whoever's actually WORKING the ticket can move it through its lifecycle without needing org-wide ticket.assign.
  • Assigning/reassigning: NO ownership exception at all - a flat requirePermission(TICKET_ASSIGN) check, full stop. Routing work to the right person is inherently a coordinator action, not something a member should be able to do to themselves or others just by asking.

See ticket.service.ts's updateTicket vs. updateTicketStatus for exactly how each one checks a different field on the same document.

What actually happens, step by step

1
Any member clicks "Tickets" in the sidebar - a brand-new nav link, right alongside Projects.
2
They file one: a title, optional description, and a priority. No permission needed - it's created immediately, unassigned, status "open."
3
Someone with ticket.assign opens it and assigns it to a teammate from a dropdown of real org members (re-checked as an ACTIVE membership, same Day-4/7/8 pattern).
4
The assignee works the ticket, moving its status from "open" to "in progress" to "resolved" - allowed because they're the assignee, even without ticket.assign themselves.
5
Back and forth happens in the comment thread - any member can reply, same low-risk "no permission gate" reasoning as task comments.
6
The original reporter tries to reassign it to someone else - gets a clean 403, since editing your own ticket's details doesn't include the power to assign it.

A design detail worth understanding: why assignment has its own endpoint

PATCH .../assign is separate from PATCH .../ticketId Assigning a ticket could have just been one more field on the general "update a ticket" endpoint - instead it's PATCH /tickets/:id/assign, its own route, gated by requirePermission(TICKET_ASSIGN) directly at the route level (no ownership check needed inside the controller at all, unlike every other write in this module). Splitting it out means the permission boundary is enforced in the SIMPLEST possible way for the one action that genuinely has no ownership exception, instead of complicating the general update endpoint with "is this specific field being changed, and if so, check a stricter rule." Same instinct as Day 8's moveTaskSchema being its own narrow schema, separate from the general task update.

What was deliberately left simple for today

  • Ticket comments can't be edited or deleted, and have no @mentions - reusing Day 9's richer comment machinery here is a reasonable follow-up, not something today's already-full scope (a whole new module) needs to also absorb.
  • No live updates via Socket.IO - tickets don't broadcast the way tasks/task-comments do since Day 9. Extending the same room-and-broadcast pattern to tickets would be a small, mechanical addition on top of what Day 9 already built, not a new concept.
  • No file attachments on tickets (unlike tasks) - not asked for, and would just repeat Day 8's already-disclosed link-based-attachment scope choice for a second model.
  • No ticket deletion - only status changes, all the way through to "closed." Same "archiving sidesteps the harder question" reasoning as Day 7's projects.
  • Priority is this day's own small design choice (low/medium/high/urgent) - not named anywhere in the SRS comments seen so far, a standard, reasonable field for a support-ticket module.

How to try it yourself

  1. Run pnpm install (no new packages today).
  2. Run pnpm dev, log in, open an organization, click "Tickets."
  3. File a ticket as a regular Member-role account if you have one (or just use your own).
  4. As an Owner/admin account, open it and assign it to someone.
  5. Log in as the assignee, move its status from open → in progress → resolved.
  6. Confirm a plain Member without ticket.assign gets a 403 trying to reassign it.
