Hardening

Valkey has sat in docker-compose.yml, completely unused, since Day 1 - even its own env var (VALKEY_URL) has been silently ignored since Day 1, the exact same gotcha Day 12 found for MinIO. Today it finally does something: backing rate limits on the routes worth attacking, alongside a render-crash safety net and a small accessibility pass.

The one-paragraph summary

"Hardening" days are grab-bags by nature - a handful of smaller, independent improvements rather than one feature, the same shape Day 3 and Day 11 already took. Today bundles three: rate limiting on login/register/forgot-password/reset-password/refresh/invitation-acceptance, backed by Valkey (closing an 16-day-old dead-code gap); a React error boundary so a genuine render-time bug shows a friendly screen instead of a blank one; and a small accessibility pass on the one modal in the app (Escape to close, click-the-backdrop to close, and aria-labels on every icon-only button that was missing one).

Same rule as always Every changed file has comments explaining what's new - especially lib/rateLimiter.ts, where the most interesting (and most surprising) part of today's work lives.

Three new concepts, explained simply

1. A THIRD "provisioned but silently ignored" env var, found the same way as Day 12's

VALKEY_URL=redis://localhost:6379 has been sitting in .env/ .env.example since Day 1, next to a real valkey service in docker-compose.yml - and nothing ever read it, because config/env.ts's Zod schema never asked for it. This is the exact same "a schema without .strict() silently ignores env vars it doesn't ask for" gotcha Day 12 found for MinIO's five env vars - by this point, a recognizable PATTERN worth actively checking for whenever a "why doesn't this setting do anything" question comes up in this codebase.

2. Fail-open, not fail-closed - and a real crash caught along the way

Rate limiting is a hardening LAYER on top of auth, not a replacement for it - so if Valkey (its storage backend) is unreachable, the correct behavior is to let requests through as if no limit existed, never to block everyone out or crash the server. Building this fail-open wrapper surfaced a genuine bug in the process: rate-limit-redis's RedisStore constructor eagerly fires two "load this Lua script" commands the moment it's created, without anyone awaiting them - when Valkey is unreachable, those two promises reject almost immediately, and with nothing ever attached to their .catch(), Node treats that as an "unhandled rejection" and crashes the ENTIRE process by default. This wasn't a hypothetical worry - running the first draft against no Valkey crashed the server outright. The fix was two no-op .catch() calls attached right after construction (see lib/rateLimiter.ts's comment on this exact spot).

3. Routing a THIRD-PARTY middleware's failures through the app's OWN error shape

express-rate-limit has its own default "too many requests" response format - but every other error in this API already flows through one place, errorHandler.ts's ApiError class, giving every response the same { success, code, message } shape no matter which route or module produced it. Today's rate limiter uses a handler callback that calls next(new ApiError(429, "RATE_LIMITED", ...)) instead of accepting the library's own default - a small thing, but it means the frontend never needs a special case just for rate-limited responses.

What actually happens, step by step

1
Someone scripts repeated login attempts against one account. Each request first passes through authRateLimiter, which asks Valkey "how many hits has this IP had in the last 15 minutes?"
2
Once that IP crosses 20 attempts in the window, the limiter calls its handler, which throws an ApiError(429, "RATE_LIMITED", ...) - the SAME error pipeline every other rejected request in the app already uses.
3
Separately: if Valkey happens to be down entirely, every one of those hit-counting attempts fails - but failOpenStore catches that failure and reports "this looks like a first-ever hit," so the request sails through instead of getting stuck or crashing.
4
On the frontend: if a genuine bug ever makes a component throw while rendering, ErrorBoundary (wrapping the entire app in main.tsx) catches it and shows "Something went wrong / Reload" instead of a blank white screen.
5
Opening a task's detail modal and pressing Escape (or clicking the dimmed backdrop) closes it - previously the only way out was clicking the small X in the corner.

A design detail worth understanding: the crash that only showed up when things went wrong

Testing the FAILURE path is what caught this bug This bug would never have been noticed by testing the happy path (a real, running Valkey) - it only exists when Valkey is UNREACHABLE, which is exactly the scenario this sandbox actually has, with no real Valkey ever available. That turned into a genuine advantage this time: unlike Day 12's MinIO (whose failure path could only be reasoned about, not run), this feature's fail-open behavior could be - and was - fully exercised and proven correct, right here, without needing any real infrastructure at all. Deliberately building and testing the DEGRADED path, not just the happy one, is the actual lesson of today's hardening work.

What was deliberately left simple for today

  • Rate limiting is per-IP only, not per-account or per-IP-plus-email - simpler, and still meaningfully slows down scripted brute-forcing. Combining both would catch a slightly wider set of attack patterns, a reasonable further hardening step.
  • No full focus-trap on the modal (Tab cycling ONLY within the dialog while it's open) - Escape and backdrop-click cover the actual missing "how do I get out of this" gap; a full trap is a further accessibility polish item, not required to fix today's gap.
  • The error boundary shows one generic message for every crash, with no per-error detail shown to the user (it's still logged to the console) - a real production deployment would send crash details to an error-tracking service instead of only the console, a reasonable further hardening step this teaching project doesn't need to prove the underlying mechanism.
  • Security headers (helmet) and input handling were reviewed, not rebuilt - Day 1's helmet config and the project's consistent use of Zod validation + React's automatic text-escaping (no dangerouslySetInnerHTML anywhere in the codebase) were already solid; nothing needed changing there today.

How to try it yourself

  1. Run pnpm install - THIS TIME there ARE new packages (express-rate-limit, ioredis, rate-limit-redis).
  2. Make sure Docker's Valkey container is running (docker compose up -d valkey if needed).
  3. Run pnpm dev, then fire 20+ rapid login attempts (wrong password is fine) - confirm the 21st gets a 429 with a RATE_LIMITED code instead of the usual 401.
  4. Stop the Valkey container and repeat - confirm login still works normally (no crash, no block) - that's the fail-open path in action.
  5. Open a task's detail modal and press Escape, then reopen it and click the dimmed area outside the panel - both should close it.
