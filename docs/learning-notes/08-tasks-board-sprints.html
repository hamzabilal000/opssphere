DAY 8 OF 18

Tasks, Board & Sprints

The actual day-to-day work surface of the app - create a task, put it on a sprint, drag it across a board, comment on it, log time, mark it done.

The one-paragraph summary

Day 7 gave OpsSphere Projects and the people on them. Today gives those projects something to DO - Tasks (with subtasks), a real Kanban board with drag-and-drop, a lightweight Sprint model, comments, link attachments, and manual time entries. The SRS explicitly defers dependencies, checklists, and a risk register to later ("should have, not today's job") - what's here is the full loop the Day 8 acceptance test asks for: create, assign, drag across the board, comment, mark done.

Same rule as always Every new file has comments explaining what's new - especially task.service.ts's moveTask, which is where the one real business rule for today lives.

Four design decisions worth understanding

1. A subtask is just a Task pointing at another Task

There's no separate "Subtask" model. Task has an optional parentTaskId - when it's set, that task is a subtask of whichever task it points to. This means subtasks get everything a normal task gets for free (their own status, assignees, comments, time entries) without building a second, parallel model that duplicates most of the same fields.

2. The board's four columns aren't in the SRS - they're a reasonable, small default

TaskStatus is "todo" | "in_progress" | "in_review" | "done". The SRS asks for "a Kanban board with drag-and-drop" but doesn't name exact columns - four is a standard, small set that supports the acceptance test's full loop without inventing workflow states nobody asked for.

3. The one real server-validated rule: can't finish a task with open subtasks

Dragging a card to "Done" calls moveTask, which checks: does this task have any subtask that ISN'T done yet? If so, it comes back with a clean 409, the same shape every other "blocked action" in this project uses (Day 5's role deletion, Day 7's milestone... actually Day 7 didn't need one - Day 4's "last owner can't demote themselves" is the closest precedent). The frontend never has to know this rule exists - the drag either succeeds or the drop bounces back with a clear message.

4. Attachments are links, not uploaded files - on purpose

A real "upload a file" feature means multipart parsing, an S3-compatible client, and wiring up the MinIO bucket that's been sitting unused in docker-compose.yml since Day 1 - a legitimate next step, but new infrastructure this environment can't actually spin up AND verify in one day. Storing a name + URL still delivers the real feature ("attach something to a task") honestly, instead of quietly skipping verification on a half-built upload pipe.

The permission split, explained

Two new permissions: task.manage (create/edit/delete/move a task) and sprint.manage (create/edit/delete a sprint). Comments, attachments, and time entries are deliberately NOT behind a permission - any active org member can post a comment, attach a link, or log their own time, the same way anyone on a real team can. Deleting someone else's comment isn't built at all yet (Day 8 keeps comments append-only); deleting an attachment or time entry is allowed for whoever created it, OR someone with task.manage - the same "ownership OR elevated permission" shape Day 3's session revocation already established.

What actually happens, step by step

1
You open a project and click the new "Board" link.
2
You add a task. It appears at the bottom of the To Do column.
3
You click the card, assign it to a project member, add a subtask, and post a comment.
4
You drag the card to In Progress, then In Review, then Done - each drop is a real API call under the hood.
5
If it still has an open subtask, the drop to Done bounces back with a clear error instead of silently succeeding.

A design detail worth understanding: why the board only shows top-level cards

Subtasks live inside the modal, not as their own column cards Every task returned by listTasks is on the board - including subtasks - but TaskBoardPage.tsx filters to only !task.parentTaskId before rendering columns. Putting both a task AND its subtasks as separate cards on the same four columns gets visually confusing fast (which card belongs to which parent?). Subtasks instead show as a short checklist inside their parent's detail modal - simpler to look at, and the "can't finish with open subtasks" rule is naturally visible right where you'd manage them.

What was deliberately left simple for today

  • Dragging a card always appends it to the END of the target column - not "insert at this exact position and shift every other card." position is a plain number set by counting existing cards in the target column; true reordering-within-a-column is a reasonable future upgrade.
  • No task dependencies, checklists, or a risk register - the SRS names these as "should have," not today's job.
  • Comments can't be edited or deleted, and there's no @mentions - that's a Day 9+ "Real-Time" feature per the SRS, not today's.
  • Subtasks only go one level deep in the board UI (a subtask's OWN subtasks, if you made any via the API directly, wouldn't get a nested checklist in the modal) - the backend doesn't actually stop deeper nesting, this is a frontend display simplification only.

How to try it yourself

  1. Run pnpm install (no new packages today).
  2. Run pnpm dev, log in, select an organization, open a project.
  3. Click "Board," add a task, click it, assign it to yourself, add a subtask.
  4. Drag the task across the columns to Done - try it WITH the subtask still open first, to see the 409 rule fire, then mark the subtask done and try again.
  5. Post a comment, attach a link, and log some time on the task from its detail panel.
