DAY 7 OF 18

Projects & Members

The project-tracking backbone — the first "real business data" in OpsSphere, everything before today was accounts and access control.

The one-paragraph summary

Days 1-6 built the scaffolding: accounts, organizations, roles, permissions, a real frontend shell. Today is the first day OpsSphere actually tracks something a company would use it for - a Project, who's on it, and its milestones. Deliberately simple: no tasks, no Kanban board, no sprints yet (that's Day 8) - just "stand up the backbone," in the SRS's own words.

Same rule as always Every file involved has comments explaining what's new - especially project.service.ts's addProjectMember, which shows how a SECOND-level membership check works, layered on top of Day 4's org-level one.

Two new concepts, explained simply

1. A second, SMALLER membership system, nested inside the first

Day 4 built Membership (user ↔ organization). Today adds ProjectMember (user ↔ project) - the exact same join-table idea, one level deeper. The key design choice: ProjectMember roles are just "lead" or "member" - a plain fixed choice, NOT another full custom-role system like Day 5's organization Roles. A project doesn't need its own permission catalog; it just needs to know who's running it. Two different problems, two appropriately different amounts of complexity.

2. You can only add people who are ALREADY in the organization

Before adding someone to a project, addProjectMember checks that they have an ACTIVE Membership in the organization first - not just any user id, a real, currently-active org member. This mirrors the exact same principle as Day 4's tenant isolation: never trust an id at face value, always re-check it against a real record. Here it stops a project from ever containing someone who isn't even part of the company running it.

What actually happens, step by step

1
You click "Projects" in the sidebar - a new nav item, only shown once an organization is selected.
2
You create a project. You're automatically added as its first "lead" - same pattern as Day 4's "whoever creates an org becomes its Owner."
3
You open the project and add two members, picking from a dropdown of the organization's OWN members (people outside the org simply can't be selected).
4
You add a milestone with a due date, and check it off once it's done.
5
Back on the projects list, the project now shows "3 members" - the Day 7 acceptance test, working end to end.

A design detail worth understanding: two permission-gated levels, one route file each

Why project.routes.ts is its own file, not more lines in organization.routes.ts Every project URL starts with /:organizationId/projects - same base path as Day 4-5's organization routes - but lives in its OWN router, mounted alongside it in app.ts. Express happily supports two separate routers sharing one prefix, as long as their more specific paths don't collide. Splitting by FEATURE (organizations vs. projects) instead of cramming everything into one growing file keeps each one focused - a pattern worth continuing as more modules (tasks, tickets, ...) get added in the days ahead.

What was deliberately left simple for today

  • Project templates are explicitly deferred by the SRS - plain creation is enough for now.
  • No project DELETE endpoint - only status changes (active/completed/archived). Deleting a project would mean deciding what happens to its members and milestones; archiving sidesteps that question entirely and is usually what you actually want anyway.
  • No tasks, subtasks, Kanban board, or sprints - all Day 8.

How to try it yourself

  1. Run pnpm install (no new packages today).
  2. Run pnpm dev, log in, select an organization.
  3. Click "Projects" in the sidebar, create one.
  4. Open it, add two members (you'll need at least two people in the organization already - invite one via the Organization page first if needed), and add a milestone.
  5. Go back to the projects list and confirm the member count shows correctly.
